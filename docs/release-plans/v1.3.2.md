# v1.3.2 Release Plan — Typed-tool depth for high-risk commands

**Target version:** 1.3.2 (patch release)
**Estimated effort:** ~1 engineer-week (~5 working days)
**Owner:** TBD
**Depends on:** v1.3.1 (registry + explainer breadth across categories 7–12, 14, 16)

---

## 1. Why this release exists

### 1.1 Where 1.3.1 leaves us

v1.3.1 closes the breadth gap from the 16-category exec audit by adding **registry aliases** (Layer 1) and **explainer patterns** (Layer 2) for ~50 commands across service management, permissions, process management, network diagnostics, scheduling, etc. Operators see plain-language HITL messages for any of these commands and policies can target them by action class.

What 1.3.1 *does not* do is replace the `bash`/`exec` execution path. The agent still calls a generic shell tool with a string command. Two consequences:

1. **Shell injection surface remains.** A command string like `systemctl restart $UNIT_FROM_ENV` passes the explainer's pattern check ("looks like a normal systemctl call"), but if the substituted value contains a shell metacharacter, the agent has just executed something the operator didn't approve. This is the same risk vector that motivated the v1.3.0 typed-tool work for `npm_install`, `docker_run`, etc.
2. **Wrong-action audit trail is structural.** When the agent calls `bash` with `command: 'systemctl reboot'`, the audit log records the *string*. Reconstructing intent ("did the agent want to reboot or restart?") requires re-parsing the command. Typed tools record `{ tool: 'systemctl_reboot', args: {} }` — semantically unambiguous.

### 1.2 What this release fixes

Add typed-tool wrappers for the highest-risk commands surfaced in 1.3.1's coverage map. Each wrapper:

- Has a TypeBox-validated manifest with explicit parameters (no free-form strings).
- Calls the real binary via `spawnSync` with an explicit argv array (no shell, no metacharacter expansion).
- Maps to the action class introduced in 1.3.1 (e.g. `system.service`, `process.signal`, `permissions.modify`).
- Inherits the action class's risk + HITL defaults from the registry.

The result: agents that explicitly choose a typed tool get a pre-validated, structured execution path. Agents that fall back to `bash` still get the v1.3.1 explainer + HITL on the command string — the typed tools are *additional* defense-in-depth, not a replacement.

## 2. Scope

### 2.1 In scope

| Workstream | Deliverable | Est. |
|---|---|---|
| W1 | `systemctl` typed tool — `unit` + `action` enum (start/stop/restart/reload/enable/disable/mask/unmask/status) | 0.5d |
| W2 | `reboot` / `shutdown` typed tools — explicit confirmation params, scheduled-time support | 0.5d |
| W3 | `chmod` / `chown` typed tools — mode/owner params with validation, recursive flag | 0.5d |
| W4 | `kill` / `pkill` typed tools — pid + signal enum, name-based variant | 0.5d |
| W5 | `kubectl` typed tool — subcommand dispatch (`apply`, `get`, `delete`, `rollout`), namespace param | 1d |
| W6 | `docker push` typed tool — image + registry params | 0.25d |
| W7 | `crontab` typed tool — list / install-from-file / remove (no inline edit — too injection-prone) | 0.5d |
| W8 | E2E regression: each tool exercised end-to-end through the enforcement pipeline | 0.5d |
| W9 | CHANGELOG, README, action-taxonomy.md | 0.25d |

**Total: ~5 engineer-days ≈ 1 week**

### 2.2 What's deferred / handled differently

- **`sudo` and `passwd`.** Skipping the typed-tool wrapper. These commands should be **blocked outright in default policy**, not approved via HITL. The 1.3.1 work registers them under a `permissions.elevate` action class with a forbid rule shipped in default rules. Operators who genuinely need them can permit via custom policy or use `unsafe_admin_exec`. Documented in §17 of the v1.3.0 plan as a deliberate posture.
- **`apt`, `yum`, `dpkg`, `snap`, `brew`.** Already covered by the existing `package.install` typed tools (`pip_install`, `npm_install`). Adding distro-specific wrappers is low-priority; defer to v1.4.
- **`kubectl exec`, `kubectl proxy`, `kubectl port-forward`.** These have shell-execution and persistent-tunnel semantics that don't fit the `spawnSync`+return pattern. Defer to a dedicated v1.4 design conversation.

### 2.3 Explicitly out of scope

- **Wrapping every command in 1.3.1's coverage map.** Most low-risk commands (`cat`, `ls`, `ps`, `df`, etc.) get nothing from a typed tool and are read-only anyway.
- **Replacing the v1.3.0 `bash`/`exec` path.** Agents that prefer free-form shell commands continue to work; the typed tools are opt-in for agents that want structured execution.
- **Dynamic param schemas.** Each typed tool has a fixed TypeBox schema. No support for "user-defined parameter shapes" — that's deliberate friction so operators can't accidentally widen the surface.

## 3. Workstream W1 — `systemctl` typed tool

### 3.1 Manifest

```ts
const SystemctlSchema = Type.Object({
  unit: Type.String({ minLength: 1, maxLength: 256, pattern: '^[a-zA-Z0-9._@-]+$' }),
  action: Type.Union([
    Type.Literal('start'),
    Type.Literal('stop'),
    Type.Literal('restart'),
    Type.Literal('reload'),
    Type.Literal('enable'),
    Type.Literal('disable'),
    Type.Literal('mask'),
    Type.Literal('unmask'),
    Type.Literal('status'),
    Type.Literal('is-active'),
    Type.Literal('is-enabled'),
  ]),
});

export const systemctlManifest: SkillManifest = {
  name: 'systemctl',
  action_class: 'system.service',
  schema: SystemctlSchema,
  // risk + hitl_mode inherited from action class via registry
};
```

### 3.2 Implementation

```ts
const result = spawnSync('systemctl', [params.action, params.unit], {
  encoding: 'utf-8',
  shell: false,
});
return { stdout, stderr, exit_code: result.status ?? 1 };
```

The `unit` regex `^[a-zA-Z0-9._@-]+$` rejects shell metacharacters at validation time. No subshell ever runs.

### 3.3 Acceptance

- TC-SCT-01..09: each action enum value executes successfully against a stub binary in CI.
- TC-SCT-10: invalid unit name (`'nginx; rm -rf /'`) is rejected with a TypeBox validation error before any spawn.
- TC-SCT-11: missing `unit` param produces a 400-style error.
- E2E: agent calls `systemctl_unit_action` → HITL fires (per_request) → operator approves → command executes → audit log shows structured `{tool, action, unit}`.

**Estimate:** 0.5d.

## 4. Workstream W2 — `reboot` / `shutdown`

### 4.1 Design

Two separate typed tools so the action_class lookup picks the right risk profile and so audit entries are unambiguous:

- `reboot` — no params other than optional `confirm: true` (must be present to execute; structural barrier against accidental triggering).
- `shutdown` — `mode: 'poweroff' | 'reboot' | 'cancel'` + optional `time: string` (validated against the regex `^now$|^\+\d+$|^\d{1,2}:\d{2}$`).

Both map to `system.service` action class.

### 4.2 Acceptance

- `reboot` without `confirm: true` is rejected at validation.
- `shutdown` with `mode: 'cancel'` cancels pending shutdowns; uses `shutdown -c` underneath.
- Invalid `time` format rejected before spawn.

**Estimate:** 0.5d.

## 5. Workstream W3 — `chmod` / `chown`

### 5.1 Manifest highlights

- `chmod`: `path: string` (file_path-validated), `mode: string` (regex `^[0-7]{3,4}$|^[ugoa]*[+-=][rwx]+(,[ugoa]*[+-=][rwx]+)*$`), `recursive?: boolean`.
- `chown`: `path`, `owner: string` (regex `^[a-z_][a-z0-9_-]*(:[a-z_][a-z0-9_-]*)?$`), `recursive?: boolean`.

### 5.2 Action class

New action class `permissions.modify` registered in 1.3.1 with default risk `high` and HITL `per_request`.

### 5.3 Acceptance

- `chmod_path { mode: '777', recursive: true, path: '/' }` — typed tool produces the exact spawn invocation; HITL message warns about recursive permission change at root.
- Symbolic mode (`+x`) accepted; arbitrary string (`'rw-r--r-- ; rm -rf /'`) rejected.

**Estimate:** 0.5d.

---

## 6. Workstream W4 — `kill` / `pkill`

### 6.1 Design

- `kill_process { pid: number, signal?: 'TERM' | 'KILL' | 'HUP' | 'INT' | 'USR1' | 'USR2' }` — `pid` is integer-validated.
- `pkill_pattern { pattern: string, signal?: ... }` — `pattern` validated against a non-shell regex (no metacharacter expansion).

Action class: new `process.signal` registered in 1.3.1 (default risk `high`, HITL `per_request`).

### 6.2 Acceptance

- `pid` of 1 (init) or 0 still HITL-gated even though they are valid integers.
- Default signal is `TERM`; `KILL` requires explicit specification.

**Estimate:** 0.5d.

---

## 7. Workstream W5 — `kubectl`

### 7.1 Design

The richest typed tool of the set because `kubectl` has dozens of meaningful subcommands. Restrict to the four highest-volume in operator workflows:

- `kubectl_apply { manifest_path: string, namespace?: string, dry_run?: boolean }`
- `kubectl_get { resource: string, name?: string, namespace?: string, output?: 'yaml' | 'json' | 'wide' }`
- `kubectl_delete { resource: string, name: string, namespace?: string, grace_period?: number }`
- `kubectl_rollout { action: 'status' | 'restart' | 'undo', resource: string, name: string, namespace?: string }`

Each is its own typed tool (clearer audit; allows different risk levels per subcommand — `get` is `medium`, `delete` is `critical`).

Action class: new `cluster.manage` (or split into `cluster.read` / `cluster.write`). Decided in 1.3.1.

### 7.2 Acceptance

- Each tool's manifest exercised end-to-end against a stub kubectl binary in CI.
- Resource-name regex rejects shell metacharacters and path-traversal sequences.
- `namespace` defaults to current context — operators don't need to specify it for the common case.

**Estimate:** 1d (largest single item in the release).

## 8. Workstream W6 — `docker push`

Trivial wrapper: `docker_push { image: string, registry?: string, all_tags?: boolean }`. Image-name regex (`^[a-z0-9._-]+(:[a-z0-9._-]+)?$` for tag syntax). Registers under `code.execute` (already exists) or a new `cluster.manage` action class TBD.

**Estimate:** 0.25d.

---

## 9. Workstream W7 — `crontab`

### 9.1 Design

`crontab` is uniquely dangerous because the inline `-e` editor mode is interactive and the file-install mode (`crontab <file>`) silently replaces the user's entire schedule. The typed-tool wrapper exposes only the **safe** operations:

- `crontab_list` — read-only listing (action class: `system.read`).
- `crontab_install_from_file { file_path: string, replace_confirm: true }` — required `replace_confirm` parameter mirrors the `confirm` pattern from `reboot`.
- `crontab_remove { user?: string }` — `crontab -r`, with explicit username so an agent can't accidentally remove the wrong user's crontab.

**No** `crontab_edit` typed tool — interactive editing is out of scope; operators who need it use `unsafe_admin_exec`.

Action class: new `scheduling.persist` registered in 1.3.1.

**Estimate:** 0.5d.

## 10. Workstream W8 — E2E regression

New tests:

- **`src/tools-systemctl.e2e.ts`** — full call → HITL → approve → spawn → result for each `action` enum value.

- **`src/tools-kubectl.e2e.ts`** — apply / get / delete / rollout against a stubbed kubectl.

## 12. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| TypeBox schema rejects a legitimate value (e.g., a unit name with a `+` character) and the agent can't proceed | Medium | Low | Conservative regexes, exhaustive test coverage of valid values. Agents fall back to `bash` + HITL. |
| `kubectl` subcommand surface grows beyond the four shipped → operators want more | Medium | Low | v1.4 adds the next four. The four shipped cover ≥80% of operator volume per audit. |
| Operators expect `crontab_edit` to work | Low | Medium | Document the omission and the `unsafe_admin_exec` escape hatch in the CHANGELOG. |
| Agent ignores typed tools and continues calling `bash` | High | Low | This is a *non*-issue: the v1.3.0/v1.3.1 explainer + HITL still fires on `bash` calls. Typed tools are a defense-in-depth bonus, not a replacement. |
| Action-class RFC process delays the release | Medium | Medium | File the RFCs alongside this plan, not at the end. SLA is 14 days; start the clock at sprint kickoff. |
| Wrapping breaks an operator's existing custom policy that targets `unknown_sensitive_action` | Low | Medium | Migration note in CHANGELOG; the typed tools route to specific action classes, so a custom rule on `unknown_sensitive_action` no longer matches them. Affected operators add a permit/forbid on the new class. |

## 14. Open questions for review

1. **Action-class naming.** `process.signal` vs `process.kill` — `signal` is more accurate (you can send `HUP`, not just kill). Recommend `process.signal`.
2. **`kubectl` action class.** Single `cluster.manage` for write + `cluster.read` for read, or one `cluster.manage` for everything? Recommend the split — get is HITL-noisy if it's `critical`.
3. **`unsafe_admin_exec` interaction.** Once typed tools cover most high-risk surface, should `unsafe_admin_exec` get a louder warning ("typed tools cover this — are you sure you need shell?")? Probably yes; small UX tweak.
4. **Default policy entries.** Should v1.3.2 ship default-rule forbids on `permissions.elevate` (sudo/passwd/su) so the "block by default" intent from §2.2 is enforced out of the box? Recommend yes — lower friction than asking every operator to write their own forbid.

## 15. Success criteria

1. Each of the 7 workstreams ships a typed tool that passes its E2E suite.
2. Tester audit shows reduced `unknown_sensitive_action` volume on the targeted commands (measured against v1.3.1 baseline).
3. All RFCs reach `implemented` status before the release tags.
4. All ~4150 unit + ~470 e2e tests stay green; new tests added.
5. Default policy ships with `permissions.elevate` forbid rule; operators must explicitly permit `sudo`/`passwd`/`su` to use them.

## 16. Rollback

The release is composed of independent typed tools. Each is gated behind its own action-class registration:

- **A single tool misbehaves:** Remove its alias from the registry — the tool falls back to `unknown_sensitive_action` + the v1.3.1 explainer/HITL path. No change to other tools.
- **A new action class causes policy regressions:** Revert the registry entry; rules targeting the class become inert (no matching action_class) but no tool is broken.
- **Full rollback:** Revert all 1.3.2 merges, republish 1.3.1. Takes ~30 min. The v1.3.1 explainer/HITL path keeps every command surfacing through HITL.

---

## 17. What this release explicitly does NOT do

- Does not provide typed tools for `sudo` / `passwd` / `su` — these are forbidden by default policy.
- Does not replace the `bash`/`exec` path — typed tools are additional, not exclusive.
- Does not auto-derive typed tools from observed exec patterns — pattern derivation lives in Approve Always (v1.3.0).
- Does not introduce dynamic / user-defined typed-tool schemas.
- Does not cover every command from the 16-category audit — only the highest-risk subset where wrong execution is destructive at the host level.
